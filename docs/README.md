## 🔄 文档更新规范

### 更新频率

| 文档类型 | 更新时机 | 负责人 |
|---------|---------|--------|
| 项目进度与状态 | 每周或重大变更时 | 项目经理/技术负责人 |
| 未完成功能清单 | 功能完成或新增时 | 产品经理/开发者 |
| API文档 | API变更时立即更新 | 后端开发者 |
| 数据库Schema | 数据库变更时立即更新 | 后端开发者 |
| 测试文档 | 新功能上线前 | 测试人员/开发者 |
| 问题记录 | 问题修复后 | 开发者 |

### 文档命名规范

- **技术文档**: 使用英文名称 (如 `backend_endpoints.md`)
- **项目文档**: 使用中文描述性名称 (如 `项目进度与状态.md`)
- **版本化文档**: 使用 `_v{version}` 后缀 (如 `database_schema_v3.sql`)
- **日期化文档**: 使用 `-YYYY-MM-DD` 后缀 (如 `已修复问题记录-2025-09-30.md`)

### 文档维护原则

1. **保持更新**: 代码变更时同步更新文档
2. **简洁明了**: 使用清晰的语言和结构
3. **版本管理**: 重大变更时创建新版本
4. **归档旧文档**: 过期文档移动到 `history/` 目录
5. **添加索引**: 在本README中添加新文档的链接
